Documentação técnica · Claude Code plugin · jul/2026

Um plugin, cinco comandos,
zero cópia manual.

O mss-spec adiciona um fluxo Spec-Driven leve ao Claude Code, por cima do plugin superpowers. Instala uma vez, vale em todo projeto, e faz o scaffolding dos arquivos que antes você copiava à mão — já carregando o padrão de infraestrutura MSIG (rede Docker, proxy, Postgres/SQL Server, Azure).

Comandos8 namespaced
Templates5 embutidos
Memória2 camadas / 2 índices
Dependênciasuperpowers
00 — Visão geral

O que o mss-spec é (e o que não é)

É um plugin do Claude Code que formaliza, como atalhos nomeados, o fluxo Spec-Driven que você já usava na mão. A disciplina (não codar sem OK, TDD, verificação, code review) não é dele — vem do plugin superpowers. O mss-spec adiciona só o que faltava: comandos memorizáveis, scaffolding automático e o padrão corporativo MSIG embutido.

5comandos
1skill
5templates
1×instala uma vez

Antes, adotar o padrão num projeto novo era copiar seis arquivos à mão, renomeando cada um para um destino diferente (um rename.txt existia só para lembrar a ordem). Agora você instala o plugin uma vez no escopo user e roda /mss-spec:kickoff — o próprio comando gera os arquivos no lugar certo.

A separação que define o desenho
Existe maquinaria ativa (comandos + skill), que vira plugin e é compartilhada; e existe arquivo por-projeto (o CLAUDE.md daquele repo, a pasta memory/ versionada com o código, o docs/AMBIENTE.md), que nenhum plugin "é" por você. A ponte entre os dois: o plugin embute os templates e o comando faz o scaffolding.
01 — Ciclo de vida

Como o Claude Code carrega tudo isso

Entender a ordem em que o Claude Code lê as coisas dissolve a maior parte das dúvidas — inclusive a de "por que não roda no chat comum".

No início de cada sessão

  1. Lê e injeta o CLAUDE.md (do projeto + pais + global) — vira instrução permanente. O CLAUDE.md pode instruir a ler um projeto.md da raiz antes de tudo, se existir.
  2. Lê o settings.json e habilita os plugins (superpowers, mss-spec).
  3. Enumera as skills disponíveis e apresenta ao modelo só name + description — o corpo carrega sob demanda.
  4. Enumera os slash commands dos plugins.
  5. Roda hooks de SessionStart (o mss-spec não usa hooks bloqueantes).

Quando você faz uma solicitação

Você digita /mss-spec:kickoff "ideia" → o conteúdo do command.md vira o prompt (com $ARGUMENTS substituído) → o comando invoca a skill brainstorming (do superpowers) → entrevista → seu OK → scaffolding dos arquivos → fim
Comando > skill (por quê)
Um slash command é gatilho explícito e determinístico: roda porque você digitou. Uma skill fica esperando disparar por relevância (a descrição casar com o pedido) — pode não disparar. Por isso o fluxo principal é feito de comandos, e a skill precedentes-msig é só uma rede de segurança complementar.
Não roda no chat comum
O mss-spec só funciona no Claude Code (CLI, app desktop do Code ou IDE), que tem acesso ao sistema de arquivos. No Claude Desktop / claude.ai (chat) não há acesso ao seu C:\ — é limite de plataforma, não de configuração.
02 — Instalação

Instalar via marketplace local

O plugin mora numa pasta do disco e é servido por um marketplace local. Numa sessão do Claude Code:

/plugin marketplace add C:/Ronaldo/_Mitsui/Python/Spec-Powers
/plugin install mss-spec@mss-local

Escolha o escopo user — assim os comandos e a skill ficam disponíveis em todos os projetos da máquina, exatamente como o superpowers. Requer o superpowers habilitado e Claude Code v2.1.140+.

Precisa de restart COMPLETO
Mudança de plugin/marketplace só carrega quando o Claude Code fecha por inteiro e reabre — nova aba ou nova conversa não recarrega. Faça Quit de verdade (inclusive o ícone da bandeja) e reabra. É o mesmo fenômeno do backend do Docker que ignora "Apply & Restart" até um Quit completo.

Registro manual — quando os /plugin não abrem

Os comandos /plugin só abrem o painel numa sessão interativa do Claude Code. Fora disso, registre editando ~/.claude/settings.json (escopo user, sem tela de trust) — mesmo formato do superpowers, trocando a fonte remota por pasta local (directory):

{
  "enabledPlugins": {
    "superpowers@claude-plugins-official": true,
    "mss-spec@mss-local": true
  },
  "extraKnownMarketplaces": {
    "mss-local": {
      "source": { "source": "directory", "path": "C:/Ronaldo/_Mitsui/Python/Spec-Powers" }
    }
  }
}

Caminho com barras normais /. A chave em enabledPlugins é <plugin>@<marketplace> (mss-spec@mss-local). Depois, restart completo.

Fallbacks garantidos

  • Auto-load sem marketplace: pasta ~/.claude/skills/mss-spec/ com o .claude-plugin/plugin.json dentro → carrega sozinha como mss-spec@skills-dir, sem editar settings.
  • Por sessão (CLI): claude --plugin-dir "C:/Ronaldo/_Mitsui/Python/Spec-Powers".
EscopoArquivoAlcance
user~/.claude/settings.jsonTodos os projetos da máquina (recomendado agora)
project.claude/settings.jsonO repositório inteiro (via git) — útil na promoção ao time
local.claude/settings.local.jsonSó sua cópia local, gitignored
Promoção pro time (depois de validar)
Trocar a source em .claude-plugin/marketplace.json de caminho local para um repositório git. Sem retrabalho de estrutura — só muda a fonte.
03 — Anatomia

Estrutura do plugin

A própria pasta Spec-Powers é o repo do plugin. Isto é o que existe em disco:

mss-spec/ (repo do plugin) ├── .claude-plugin/ │ ├── plugin.json manifesto (name: mss-spec) │ └── marketplace.json marketplace local (mss-local) ├── commands/ │ ├── kickoff.md nova-feature.md ambiente.md │ ├── banco.md precedentes.md plano-teste.md │ └── modo.md ├── skills/ │ └── precedentes-msig/SKILL.md catálogo de auto-disparo ├── templates/ embutidos; os comandos copiam pro projeto │ ├── CLAUDE.md settings.json MEMORY.md INDEX.md │ ├── AMBIENTE.md FRONTEND.md (design system MSIG) │ ├── docker/ docker-compose.yml · docker-compose.office.yml · Dockerfile · dockerignore │ ├── doc/template.html (estilo editorial p/ documentação) │ ├── certs/corp-ca.pem (CA corporativa MSIG) │ └── assets/logo.png (logo MSIG) └── docs/ manuais (não vão pros projetos) ├── LEIA-ME.md · ROTEIRO-SPEC-DRIVEN.md · referencia-spec-driven.md ├── COMO-FUNCIONA.html (este documento) └── superpowers/specs · plans
Templates ≠ arquivos do plugin
Os arquivos em templates/ não são a configuração do plugin — são moldes que os comandos copiam para dentro de cada projeto. O comando referencia-os via a variável ${CLAUDE_PLUGIN_ROOT}/templates/….
Comandos

Os oito comandos, um a um

Todos são namespaced (/mss-spec:<nome>) e marcados com disable-model-invocation: true — só você os dispara, o modelo não os invoca sozinho.

C1/mss-spec:kickoffconstitui o projeto

O que faz: constitui um projeto (novo ou existente). Invoca brainstorming, faz scan se for brownfield, entrevista você uma coisa por vez e, ao final, faz o scaffolding.

Escreve
CLAUDE.md, .claude/settings.json, memory/MEMORY.md, docs/superpowers/INDEX.md, docs/AMBIENTE.md e — se tiver UI — docs/FRONTEND.md + static/img/logo.png
Garante
a dependência do superpowers (verifica e, se faltar, habilita no settings)
Base
o CLAUDE.md gerado instrui ler projeto.md da raiz antes de tudo, se existir (opcional por projeto)
Quando usar
uma vez, ao adotar o padrão num projeto
scaffoldingscan brownfield

Uso

/mss-spec:kickoff "em 1 frase o que é o projeto e pra quem"

C2/mss-spec:nova-featureabre feature spec→tasks

O que faz: abre uma feature no fluxo Spec-Driven: brainstorming → Critérios de Aceite → seu OK → writing-plans → tasks pequenas → executa uma por vez (TDD + verificação).

Mantém
acrescenta 1 linha em docs/superpowers/INDEX.md ao abrir; muda status para fechada ao concluir
Git
NÃO cria branch nem worktree — isolamento é pedido explicitamente
Quando usar
por feature (feat). Bugfix/refactor/chore não precisam
mantém o índicebranch explícita

Uso

/mss-spec:nova-feature exportar-relatorio

C3/mss-spec:ambienteinfra no padrão MSIG

O que faz: copia a infra Docker padronizada (compose base + office, Dockerfile, CA corporativa, .dockerignore) e explica como subir. Proxy/TLS em 4 camadas independentes. Mostra e confirma antes de gravar.

Copia
docker-compose.yml + docker-compose.office.yml + Dockerfile + .dockerignore + certs/corp-ca.pem (padrões do plugin)
4 camadas
pull da base = daemon/Docker Desktop · build = build.args/PIP_PROXY · runtime = .env · TLS = CA embutida na imagem + SSL_VERIFY=false
Subir
casa (VPN): docker compose up -d --build · escritório: … -f docker-compose.office.yml …
compose base+office4 camadas proxy/TLSCA + SSL_VERIFY

Gotcha

load metadata … i/o timeout = pull da base (é do daemon, não usa build.args/.env). Resolva com proxy no Docker Desktop ou docker pull python:3.14-slim (cache por máquina — costuma resolver). psycopg/TCP ignora proxy.

C4/mss-spec:bancoconexão a banco

O que faz: gera o boilerplate de conexão no padrão MSIG. Pergunta SQL Server ou Postgres; para SQL Server, modo simples ou Fernet. Mostra o código e confirma antes de gravar.

Escreve
o módulo de conexão (get_connection.py) + entradas no .env.example
Nunca
gera .env real nem valores de credencial — só placeholders
SQL Server / Postgres.env.example

Variantes

Postgres via psycopg2 (PG_CONN_STR). SQL Server via pyodbc — simples (env por banco) ou endurecida (Fernet: ENCRYPTION_KEY/ENCRYPTED_CONN).

C5/mss-spec:precedentesconsulta catálogo

O que faz: consulta explícita ao catálogo de precedentes entre projetos MSIG — "isso já foi resolvido em outro projeto?". A skill homônima também dispara sozinha quando relevante.

Escreve
nada — só lê o catálogo e aponta projeto, caminho e abordagem
Regra
o catálogo é índice; sempre abra o código real de lá antes de replicar
consulta+ skill auto

Uso

/mss-spec:precedentes RAG · /mss-spec:precedentes "extração de PDF"

C6/mss-spec:plano-testebaseline anti-regressão

O que faz: roda o plano de teste base (pytest) — a rede anti-regressão ("se isso passa 100%, nada quebrou"). Se a suíte passa inteira, grava/atualiza o baseline; se não, reporta as falhas e mantém o baseline anterior.

Baseline
docs/superpowers/PLANO-TESTE.md — o comando de rodar, o que cobre, e o carimbo do último 100% (data + commit)
Regra
só vira baseline com a suíte 100% verde; o anterior é substituído (o git guarda o histórico)
Disciplina
o CLAUDE.md manda rodar antes de declarar qualquer coisa pronta — feature nova ou alteração
pytest100% ou nada

Uso

/mss-spec:plano-teste

C7/mss-spec:modonível de cerimônia

O que faz: alterna o quanto do ritual Spec-Driven é aplicado — o antídoto pra lentidão do fluxo completo. Vale da chamada em diante, até trocar. Padrão: médio.

mínimo
sem spec/plano; 1-2 perguntas; executa direto com TDD leve (ajuste pequeno/claro)
médio
design curto + plano curto + execução inline, sem subagentes nem dupla revisão
alto
ritual completo: spec + writing-plans + subagent-driven-development (feature grande/crítica)
Inegociável
em todo nível: OK antes de codar, TDD e verificação (rodar + colar saída) — só o peso do planejamento varia
3 níveispadrão médio

Uso

/mss-spec:modo mínimo · /mss-spec:modo alto. Ortogonal ao effortLevel do settings.json (padrão medium) — juntos controlam velocidade × profundidade.

C8/mss-spec:documentacaodoc HTML editorial

O que faz: gera documentação HTML no estilo editorial MSIG — este mesmo layout que você está lendo. Copia o template padrão e substitui só o conteúdo, mantendo a `<style>` intacta.

Base
templates/doc/template.html (mesmo CSS/fontes/classes); exemplo completo = este documento
Blocos
hero · section · callout · node/card · table · flow · scorebar · glossary · foot
Saída
arquivo único self-contained em docs/<nome>.html — a exceção à regra de separar CSS/JS (doc portável)
estilo fixoself-contained

Uso

/mss-spec:documentacao "arquitetura do painel"

04 — Conhecimento persistido

Memória em duas camadas, dois índices

O objetivo é não lotar o contexto: cada coisa no seu arquivo, um índice barato lido no início, e o resto aberto só sob demanda.

CamadaOndeGranularidadeÍndice
Aprendizadosmemory/por-tópico (vale pro projeto inteiro)memory/MEMORY.md
Tarefasdocs/superpowers/specs + planspor-tarefadocs/superpowers/INDEX.md

Comportamento

No início da sessão, o CLAUDE.md instrui ler só os dois índices (1 linha por item). Os arquivos pesados abrem sob demanda, quando o índice aponta relevância. Grep/Glob sobre docs/ é o fallback.

Por que dentro do repo (e não em ~/.claude)
A memória mora em memory/ dentro do repositório, versionada com o código. A memória nativa do Claude Code (~/.claude/projects/<proj>/memory/) fica presa à máquina, não viaja com o código e fragmenta entre git worktrees (é indexada por caminho absoluto). No repo, ela acompanha o projeto.

Manutenção

Quando o índice inchar, rode a skill anthropic-skills:consolidate-memory para mesclar duplicatas e podar entradas mortas — evita que a memória vire um segundo sistema para manter.

05 — Padrão corporativo

O ambiente MSIG embutido

Estes fatos foram levantados varrendo 7 projetos reais e vivem em templates/AMBIENTE.md (copiado pro projeto pelo kickoff). São idênticos entre projetos — por isso são fixos, não placeholders.

ItemValor fixo
Rede Dockermitiai_network (externa, compartilhada; docker network create mitiai_network)
Proxy corporativohttp://10.170.200.120:8080 (host: pip/setx/Docker Desktop, sem cert)
Postgrescontainer postgres-db, porta 5432, imagem pgvector/pgvector:pg15, banco miti_ai_<projeto>
SQL Serverhost MSSQLD0 = 10.170.210.36, porta 1433, via pyodbc
Azure deployACR mssaicontainerregistry; recursos mss-miti-ai-<projeto>-<amb>[-br]; RG-MSSAI-DEV (dev) / RG-MSSAI-PRD (main)
TimezoneAmerica/Sao_Paulo
Host vs container (proxy)
No host (dev local, pip install) vale a config de proxy simples — sem certificado, porque o Windows já confia na CA corporativa. Dentro de container atrás do proxy do escritório, a CA precisaria ser injetada — e isso está fora do escopo do plugin por decisão de projeto.
06 — Na prática

O fluxo de trabalho, ponta a ponta

1. Constituir /mss-spec:kickoff "ideia" → entrevista → scaffolding (CLAUDE.md, memory, INDEX, AMBIENTE) 2. Por feature /mss-spec:nova-feature <nome> → Critérios de Aceite → seu OK → plano → tasks → TDD (teste → código → roda → cola saída) → repete → review → fecha → status "fechada" no INDEX 3. Infra/banco /mss-spec:ambiente (docker-compose + proxy) /mss-spec:banco (get_connection.py) Atalhos /mss-spec:precedentes <assunto> (já existe em outro projeto?)
Mudança pequena não precisa de cerimônia
bugfix → o teste que reproduz o bug é a spec; corrige; verifica. refactor → o gate é "testes continuam verdes". chore/docs → sem spec. Só feat passa pelo nova-feature completo.

Sessão ≈ feature

O plan file (em docs/superpowers/plans/) é o estado durável que desacopla trabalho de sessão. Regra prática: uma feature por sessão; sub-tarefas da mesma feature ficam juntas; feature diferente e não-relacionada → sessão nova (higiene de contexto); bug que quebra a tarefa atual → mesma sessão.

07 — Rede de segurança

A skill precedentes-msig

Diferente dos comandos, a skill dispara sozinha quando você vai implementar algo que pode já existir em outro projeto — antes de desenhar do zero.

TópicoProjeto de referênciaObservação
RAG / busca vetorialIA Bot Agent\…\_opcao4usa pgvector no Postgres compartilhado. Não replicar o Chromadb antigo (obsoleto)
Extração de PDF + multi-LLMIA Jeday Cosseguro\AzurePyMuPDF + prompts Gemini/GPT; config por cliente com fallback
Ambiente/infra corporativaplugin mss-specver templates/AMBIENTE.md
É índice, não implementação
O catálogo aponta "olhe aqui" — antes de replicar, abra o código real do projeto referenciado, porque ele pode ter evoluído desde a última atualização da tabela. Cresce por linha, sem cerimônia.
08 — Operação

Como verificar que funciona

Os comandos /plugin são interativos — rodam numa sessão real do Claude Code, não headless. Roteiro de aceite:

  1. Registrar: /plugin marketplace add C:/Ronaldo/_Mitsui/Python/Spec-Powers
  2. Instalar (escopo user): /plugin install mss-spec@mss-local
  3. Em sessão nova, digitar /mss-spec: lista os 8 comandos; a skill precedentes-msig aparece.
  4. Numa pasta vazia: /mss-spec:kickoff "teste" gera os 5 arquivos.
  5. Testar os demais: nova-feature, banco, ambiente, precedentes.
  6. Confirmado, remover a skill pessoal duplicada: rm -rf ~/.claude/skills/precedentes-msig
Ponto "só sabendo tentando"
Se no passo 4 os templates não forem copiados, é a variável ${CLAUDE_PLUGIN_ROOT} não resolvendo na sua versão. O fallback é embutir o conteúdo dos templates dentro dos comandos — correção rápida.
09 — Roadmap

Dívidas & evolução

  • Caminhos absolutos na skill. O catálogo precedentes-msig usa caminhos C:\Ronaldo\…. Funciona solo; na promoção ao time via git, migrar para caminhos relativos ou uma convenção portável.
  • Marketplace local → git. Quando validado, trocar a source do marketplace.json para um repositório git e compartilhar com o time.
  • Fallback do ${CLAUDE_PLUGIN_ROOT}. Se a variável não resolver, embutir os templates nos comandos.

Fora de escopo (por decisão)

Hooks bloqueantes; docs/ESTADO.md/ROADMAP.md; gate AC→teste em CI; injeção de certificado corporativo; qualquer coisa que o superpowers já force via skills.

GL — Glossário

Termos, sem mistério

Pluginpacote de comandos/skills instalado uma vez
Diretório com .claude-plugin/plugin.json que empacota commands, skills, agents e hooks. Habilitado no settings.json, fica disponível em todo projeto. O superpowers e o mss-spec são plugins.
Slash commandatalho que você digita
Arquivo .md em commands/ cujo conteúdo vira o prompt. Namespaced pelo plugin (/mss-spec:kickoff). Gatilho determinístico — roda porque você digitou.
Skillinstrução que auto-ativa por descrição
Arquivo SKILL.md com frontmatter (name, description). O Claude Code enumera as descrições no boot e carrega o corpo quando o pedido casa. Pode não disparar — por isso o fluxo principal usa comandos.
Marketplace localfonte de plugins numa pasta
marketplace.json que aponta para o(s) plugin(s). Registrado com /plugin marketplace add. Aqui aponta para caminho local; na promoção ao time, para um repositório git.
Scaffoldinggerar arquivos a partir de templates
Ação do kickoff de criar CLAUDE.md, memory/, INDEX.md e AMBIENTE.md no projeto, copiando de templates/. Substitui a antiga cópia manual de arquivos.
superpowersplugin de disciplina (dependência)
Plugin oficial que fornece as skills de processo (brainstorming, TDD, verificação, debugging, code review). O mss-spec depende dele — o kickoff garante que esteja habilitado.
papel de especialistapersona sênior por domínio
Regra no CLAUDE.md gerado: para cada tarefa, o assistente assume o papel de especialista sênior daquele domínio (UI/UX, desenvolvimento, DBA, segurança, infra…), anuncia o papel, trabalha sob essa ótica até o OK e volta ao papel padrão (arquiteto/engenheiro sênior de desenvolvimento). Nasceu de entregas de layout genéricas ruins que melhoraram ao pedir "especialista sênior em UI".
projeto.mdbase do projeto, lida antes de tudo
Arquivo opcional na raiz do projeto com o contexto/base fundamental. O CLAUDE.md gerado pelo kickoff instrui lê-lo primeiro, no início de toda sessão, se existir. Como o CLAUDE.md é sempre carregado, é o mecanismo de "ler antes de tudo" — e o "se existir" evita erro nos projetos que não têm.
${CLAUDE_PLUGIN_ROOT}caminho do plugin em runtime
Variável que os comandos usam para referenciar arquivos embutidos no próprio plugin (ex.: ${CLAUDE_PLUGIN_ROOT}/templates/CLAUDE.md). Se não resolver, o fallback é embutir o conteúdo no comando.
