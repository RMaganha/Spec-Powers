Guia para todos · técnicos e não-técnicos · jul/2026

Um assistente que já sabe o jeito da casa.

O mss-spec é um plugin criado pelo Ronaldo Maganha (MSIG) que trabalha em conjunto com o Claude Code — o assistente de programação da Anthropic. Ele ensina o assistente a começar cada projeto já do jeito certo da empresa: com organização, checagens de segurança e o padrão de infraestrutura da MSIG prontos, sem ninguém precisar lembrar de tudo à mão.

00 — O que é

Em uma frase, para todos

Imagine um desenvolvedor novo muito rápido, mas que não conhece as manias da empresa. O mss-spec é o manual de boas práticas que você entrega a ele uma vez — e a partir daí ele já sabe onde cada coisa fica, como a MSIG monta seus sistemas, e não pula etapas importantes.

Tecnicamente: é um plugin que adiciona atalhos ao Claude Code. A disciplina de trabalho (não sair fazendo sem alinhar, testar, verificar, revisar) não é invenção dele — vem de outro plugin, o superpowers. O mss-spec junta a esse rigor o jeito da MSIG: comandos fáceis de lembrar, geração automática de arquivos e o padrão de infraestrutura da casa já embutido.

O que ele NÃO é
Não é a inteligência artificial em si (essa é o Claude Code), nem substitui o desenvolvedor. Ele é o combinado da equipe transformado em atalhos — para o assistente e a pessoa trabalharem juntos sem reinventar a roda a cada projeto.
Mais que atalhos: um jeito de trabalhar
No fundo, o mss-spec é um framework opinativo: ele impõe uma forma de desenvolver — mas uma forma baseada em boas práticas, pra ninguém decidir tudo do zero a cada vez. É a mesma filosofia de ferramentas conhecidas de quem é da área (Ruby on Rails, Angular CLI, Create React App, Yeoman), que embutem uma opinião sobre como fazer as coisas. A diferença: aqui a opinião é voltada ao desenvolvimento assistido por IA e ao contexto da MSIG.
01 — O exemplo

Um projeto real, do começo ao fim

Pra não falar no abstrato, este guia acompanha um sistema de verdade — do primeiro comando à publicação. Toda vez que um comando aparecer, você vê o exemplo aplicado nele.

O sistema que vamos construir
Um CRUD de cadastro de clientes — nome, endereço, telefone, e-mail e WhatsApp — que, numa etapa seguinte, faz um disparo automático de mensagem no WhatsApp do cliente. É um caso comum de TI: guardar dados e agir sobre eles.

Quem é quem

Três peças trabalham juntas — e entender o papel de cada uma dissolve quase toda dúvida:

Claude Code — o desenvolvedor que efetivamente escreve o código do cadastro de clientes (lê e edita os arquivos, roda os testes).
superpowers — a disciplina desse desenvolvedor: não sai codando sem alinhar, escreve teste antes, verifica rodando de verdade, pede revisão.
mss-spec (feito pelo Maganha) — a camada MSIG: transforma "quero um cadastro de clientes" num projeto já no padrão da casa (pastas em camadas, conexão ao banco segura, infra Docker, checagens de segurança) sem ninguém copiar arquivo à mão.

Por baixo, o encaixe é simples:

  1. Ao abrir, o Claude Code lê as instruções permanentes do projeto e liga os plugins.
  2. Ele lista os atalhos disponíveis (os "/comandos").
  3. Você digita um atalho (ex.: /mss-spec:kickoff) e ele executa aquele roteiro — de forma previsível, porque você pediu.
Comando > sugestão automática
Um comando roda porque você digitou — é garantido. Uma sugestão automática (skill) só dispara se o assistente "achar" que faz sentido — pode não vir. Por isso o fluxo principal do kit é feito de comandos, que você controla.
Onde funciona
Só dentro do Claude Code (o app/CLI que mexe nos seus arquivos). No chat comum do site/app (claude.ai) não funciona — lá a IA não tem acesso aos arquivos do seu computador. É limite da plataforma, não do kit.
02 — Instalação

Instala uma vez, vale sempre

O plugin mora numa pasta e é servido por uma "lojinha" (um marketplace). O mesmo marketplace.json serve às duas vias abaixo — o Claude Code resolve o plugin na raiz do marketplace, seja ele adicionado por URL git (que ele clona) ou por pasta local. Escolha o escopo user: os comandos valem em todos os seus projetos. Requer Claude Code v2.1.140+.

Via git (para o time)

Instala e atualiza sem copiar pasta. Numa sessão do Claude Code:

/plugin marketplace add https://github.com/RMaganha/Spec-Powers.git
/plugin install mss-spec@mss-local
  • https://github.com/RMaganha/Spec-Powers.git é o endereço do repositório (público no GitHub). Dá pra fixar tag/branch com #ref (ex.: https://github.com/RMaganha/Spec-Powers.git#v0.10.0).
  • Atualizar quando o autor publica versão nova: /plugin marketplace update mss-local — dá git pull no clone e re-resolve o plugin.

Via pasta local (para dev/teste)

Sem git, ou pra testar o kit em desenvolvimento: extraia/clone o repositório numa pasta local qualquer (ex.: C:/ferramentas/mss-spec, sem espaços/acentos) e aponte o marketplace pra ela:

/plugin marketplace add C:/ferramentas/mss-spec
/plugin install mss-spec@mss-local

Só o caminho antes de mss-local muda de máquina pra máquina. (mss-local é o nome fixo da lojinha.)

O superpowers é pré-requisito (passo à parte)
O mss-spec depende do superpowers (a disciplina de trabalho). Ele não é instalado junto automaticamente — habilite-o à parte (é o plugin oficial da Anthropic, superpowers@claude-plugins-official). Ao rodar o /mss-spec:kickoff num projeto, o kit já deixa o superpowers ligado nas configurações daquele projeto.
Precisa fechar e reabrir de verdade
Instalar/atualizar plugin só vale quando o Claude Code é fechado por inteiro e reaberto — abrir uma aba nova não basta. (É o mesmo drama do Docker que ignora "Apply & Restart" até um Quit completo.) Requer Claude Code v2.1.140+.
Publicado no GitHub
O kit está publicado no GitHub (github.com/RMaganha/Spec-Powers, repositório público): o time instala por URL e puxa atualizações pelo marketplace update. A via de pasta local acima continua servindo pra dev/teste.
03 — Por dentro

O que tem dentro da caixa

Você não precisa mexer nisso, mas ajuda a saber que o kit tem três tipos de coisa:

PeçaO que é
ComandosOs 19 atalhos /mss-spec:… que você digita.
Moldes (templates)Arquivos-modelo prontos (padrão MSIG) que os comandos copiam pro seu projeto — em vez de você copiar à mão.
SkillUma rede de segurança que se ativa sozinha (catálogo de "isso já foi feito em outro projeto?").
Moldes ≠ o seu projeto
Os moldes são fôrmas. Quando você roda um comando, ele copia a fôrma pro seu projeto e você preenche. Mexer nos moldes não bagunça projetos já feitos — cada projeto tem a sua cópia.
04 — Os comandos

Os dezenove atalhos, na ordem de uso

Todos começam com /mss-spec: e só você os dispara (o assistente não sai executando sozinho). Estão agrupados na ordem em que costumam aparecer num projeto — do primeiro comando à publicação. Cada cartão traz um exemplo no nosso sistema de clientes.

Começar

C1kickoffdá a largada

Prepara um projeto do zero, já no padrão da casa. É o primeiro comando que você roda.

Entrevista você (uma pergunta por vez) e depois faz o scaffolding: cria as instruções do projeto, as pastas em camadas e os arquivos-base (segurança, decisões, memória). Também deixa o superpowers ligado.

Exemplo
/mss-spec:kickoff "sistema de cadastro de clientes com nome,
   endereço, telefone, e-mail e whatsapp"
gera os arquivosentende projeto que já existe
C2doctorcheck-up do ambiente

Um "check-up" rápido do ambiente antes de começar — pega o erro cedo, não na hora de publicar.

Confere os pré-requisitos da MSIG (o proxy responde? o certificado está lá? o driver do banco existe? o superpowers está ligado? o próprio kit acha seus moldes?) e dá um veredito ✓/✗. Só avisa, não conserta. Roda quando você chama e sozinho, no começo da 1ª tarefa. Rodar de novo nunca muda nada.

Exemplo
/mss-spec:doctor
  → ✓ proxy · ✓ CA · ✗ ODBC Driver 17 (instalar antes de conectar no banco)
manual + automáticosó reporta

Montar a base

C3ambientea infra da casa

Monta a "caixa" onde o app roda, já configurada pra rede da MSIG.

Copia a infra Docker padronizada (a CA da empresa já embutida) e explica como subir. Trata o proxy e o certificado (CA) da empresa em camadas.

Exemplo
/mss-spec:ambiente
  → gera o docker-compose do sistema de clientes na rede mitiai_network
Docker padrãoproxy/TLS
C4bancoconexão ao banco

Cria o código que conecta o app ao banco de dados, sem deixar senha exposta.

Pergunta SQL Server ou Postgres. Para SQL Server, oferece guardar a senha como variável de ambiente (recomendado, mais seguro) ou via Fernet. O Fernet fica só por continuidade com projetos que já o usam — é escolha consciente do desenvolvedor, documentada aqui para ciência, não um caminho recomendado (a variável de ambiente é sempre a preferida).

Exemplo
/mss-spec:banco
  → conexão pra tabela de clientes (senha só na variável de ambiente, nunca no código)
SQL Server / Postgressegredo protegido
C5logregistro de eventos · NOVO

Liga o "diário de bordo" do app — as mensagens que contam o que está acontecendo (e o que deu errado).

Instala o padrão de log da casa: as mensagens vão pra saída padrão (que a nuvem Azure captura em produção) e também pra um arquivo local só em desenvolvimento (com ícones por nível, pra ler rápido no terminal). Você escolhe quais arquivos ganham log — nem tudo precisa. Regra dura: nunca grava senha nem dado pessoal (telefone, e-mail do cliente).

Exemplo
/mss-spec:log services/whatsapp.py
  → liga o log no serviço de disparo (registra "enviado/falhou", nunca o número)
stdout + arquivo em devnunca PII/segredoopt-in por arquivo

Construir

C6nova-featureabre uma funcionalidade

O jeito organizado de criar algo novo — combina o que vai fazer antes de fazer.

Fluxo Spec-Driven: alinha o objetivo e os critérios de aceite → seu OK → plano → tarefas pequenas com TDD. Guarda a "spec viva" do assunto e, ao fechar, roda o release e registra decisões importantes. Abre uma branch própria antes de mexer no código — sempre a partir da linha principal, nunca de outra branch.

Exemplo
/mss-spec:nova-feature "disparo automático no whatsapp do cliente"
  → critérios de aceite → seu OK → branch (da principal) → testa → verifica → revisa
spec viva por assuntobranch da principalsegurança em rota nova
C7frontendtelas ricas

Instala as ferramentas para telas mais elaboradas (grades, muitos campos).

Traz o front moderno (React + Mantine) com o tema MSIG. Telas simples ficam no jeito leve (servidor + Tailwind); telas densas ganham esse reforço.

Exemplo
/mss-spec:frontend
  → a tela de listagem de clientes (grade com busca por coluna, filtros)
React + Mantine
C8precedentes"já fizemos isso?"

Consulta um caderninho de "isso já foi resolvido em outro projeto da MSIG".

Aponta qual projeto já fez algo parecido (busca por documentos, extração de PDF, conexão a banco, integrações) para você reaproveitar em vez de refazer.

A fonte de verdade é o código, não o caderninho. O catálogo é só um índice — uma lista de "olhe aqui" —, mantido à mão: cresce uma linha por vez quando alguém acha um padrão digno de nota (a regra de como crescer está no próprio arquivo do catálogo). Por isso ele nunca se dá por atualizado — cada entrada manda você abrir o código real do projeto indicado, que pode ter evoluído desde a última anotação. É escolha de design, não descuido: como no git é o rollback, um apontador leve vale mais que uma cópia que envelhece e passa a mentir.

Exemplo
/mss-spec:precedentes "integração whatsapp"
  → "algum projeto já disparou WhatsApp? qual arquitetura usou?"
  → aponta o projeto + caminho; você abre o código de lá (a verdade)
índice, não a verdadecurado à mão, por design

Apoio no dia a dia

C9modoquanto ritual

Regula a "quantidade de burocracia" — rápido para ajuste pequeno, completo para coisa grande.

Três níveis: mínimo (vai direto), médio (padrão) e alto (ritual completo). Testar e verificar são inegociáveis em todos.

Exemplo
/mss-spec:modo alto
  → mais ritual pro disparo em massa (crítico); "mínimo" pra um ajuste de rótulo
3 níveispadrão: médio
C10memorymemória do projeto · NOVO: captura

Guarda os aprendizados e as decisões do projeto junto com o código — inclusive as decisões de não fazer algo, pra ninguém repropor semanas depois o que já foi descartado.

Tem dois modos. resgatar traz pra dentro do repositório a memória que ficou numa pasta "volátil" do Claude e a torna durável (versionada com o código). capturar (novo) destila a conversa ao fechar um assunto e distribui pros lugares certos: as decisões transversais, o que se decidiu não fazer, e um diário de sessão datado — um resumo curto do que se conversou e, sobretudo, dos pivôs (o que se cogitou, por que mudou de ideia, pra onde foi). Sempre com o seu OK antes de gravar; trechos marcados <private> nunca viram memória.

O diário mora em memory/sessions/ com um índice datado (memory/DIARIO.md) — barato de reler: abre-se só a entrada do dia/assunto, nunca a pasta toda. O fecho da nova-feature já chama o capturar sozinho; um gatilho automático opcional (desligado por padrão) pode lembrar de rodá-lo em conversas longas.

Exemplo
/mss-spec:memory resgatar
  → guarda no repo a memória volátil do Claude (não-destrutivo)
/mss-spec:memory capturar
  → decisões (incl. "não fazer") + diário da sessão, com seu OK
anti-perdadecisões + diário · NOVOanti-re-litígio
C11to-dolistbloco de recados

Um bloco de recados pessoal que aparece em qualquer parte do trabalho.

Surgiu uma ideia fora do que você está fazendo agora? Em vez de sair do foco, anote: adicionar, veja tudo com listar, e o item sai quando feito. Fica visível em qualquer branch e é só seu (não vai pro compartilhamento). É também o destino da regra um assunto por janela: quando um segundo tema aparece no meio de uma tarefa, o assistente alerta e sugere anotá-lo aqui para tratá-lo depois, numa nova conversa.

Exemplo
/mss-spec:to-dolist adicionar "enviar e-mail de boas-vindas ao cliente"
/mss-spec:to-dolist adicionar "saber qual cliente leu a mensagem"
em qualquer branchpessoalum assunto por janela

Verificar & publicar

C12plano-testerede anti-quebra

Roda a bateria de testes e guarda um "retrato do que passava" — a rede que avisa se algo quebrar depois.

Como faz: executa os testes do projeto (se ainda não há suíte, cria um mínimo — o app sobe, as rotas respondem 200, o banco conecta).
Como guarda: quando passa 100%, grava um baseline em docs/superpowers/PLANO-TESTE.md — o comando pra rodar, a lista do que os testes cobrem (1 linha cada) e um carimbo com data + commit do último verde.
Como evolui: a cada novo 100% o baseline é reescrito por cima (o git guarda as versões antigas); os testes de cada funcionalidade nova entram nele e passam a ser vigiados também.
Como compara: antes de dizer "pronto", roda-se a suíte e confere-se contra o baseline — se algo que passava agora falha, você quebrou sem querer; e o baseline só é reescrito quando tudo volta ao verde. Nunca se promove a "referência boa" algo que não passou inteiro.

Exemplo
/mss-spec:plano-teste
  → roda os testes do CRUD de clientes
  → 100% verde?  grava o baseline em PLANO-TESTE.md (data + commit)
  → algum vermelho?  reporta o que falhou e NÃO atualiza o baseline
baseline versionadocarimbo data + commit100% ou nada
C13segurancaauditoria de segurança

Passa um pente-fino de segurança no app e aponta o que corrigir.

Audita contra os riscos mais comuns (OWASP) e o baseline da casa; corrige com o seu OK. Nunca declara "está seguro" (segurança é contínua).

Exemplo
/mss-spec:seguranca
  → confere os endpoints que expõem dados do cliente (PII/LGPD, autorização)
OWASPcorrige com OK
C14complianceauditoria de convenção · NOVO

Confere se o projeto ainda está "no jeito da casa" — a estrutura e os padrões, não o funcionamento.

Com o tempo, um projeto deriva do padrão. O compliance checa a aderência às convenções do mss-spec: pastas em camadas (não tudo jogado numa pasta só), documentação e memória no lugar, segredo protegido no .gitignore. Só reporta ✓/✗ — quem conserta arquivos desatualizados é o upgrade. É o terceiro de um trio de auditorias: doctor (ambiente) · seguranca (segurança) · compliance (convenção).

Exemplo
/mss-spec:compliance
  → "o projeto de clientes segue as camadas e tem docs/memória no lugar?"
convençãosó reporta
C15releaseinspeção final · NOVO

A "inspeção final" antes de publicar: passa por tudo que importa e marca os pontos num veredito só.

É um portão de pré-publicação. Num passo, ele: roda os testes (têm que estar 100% verdes), confere versão e CHANGELOG, aplica o checklist do compliance (convenção), lembra da segurança se você mexeu numa porta de entrada, e vê se as specs batem com o entregue. Só reporta o que falta — não publica nem conserta. Verde → você segue pro merge/PR.

Exemplo
/mss-spec:release
  → ✓ testes · ✓ CHANGELOG · ✓ convenção · ✗ rodar seguranca (mexeu em rota)
junta os checkssó reportagate antes do merge

Manter & compartilhar

C16upgradeatualiza o projeto

Quando o kit melhora, atualiza um projeto antigo sem apagar o que você personalizou.

Compara os arquivos do projeto com a versão nova dos moldes: os de referência atualiza sozinho; as instruções personalizadas (CLAUDE.md) ele mescla (acrescenta o novo, mantém o seu) e só pergunta em caso de conflito real; código ele nunca mexe sozinho.

Antes de aplicar, dá pra espiar: /mss-spec:upgrade --dry-run mostra o diff do que seria mudado — inclusive nos arquivos de referência, o passo que o upgrade faria sozinho — sem escrever nada. Gostou do que viu? Roda de novo sem a flag pra aplicar. É a prévia que tira o "sobrescreveu e eu nem vi".

Exemplo
/mss-spec:upgrade --dry-run
  → mostra o diff dos moldes que mudariam (nada é escrito)
/mss-spec:upgrade
  → traz os moldes novos (ex.: o padrão de log) pro projeto de clientes já existente
mescla, não sobrescreveprévia com --dry-runconflito pergunta

Exemplo

Contexto & memória (anti-amnésia)

C18mapaonde eu parei?

Mantém um "mapa de contexto" curto do projeto — pra quem chega (ou volta) não perder tempo redescobrindo onde as coisas estão.

Guarda em docs/superpowers/MAPA.md três coisas: onde estamos (branch e assunto), o próximo passo, e as conexões com outros projetos MSIG. O assistente lê isso na partida, antes de qualquer outra coisa; o comando reconcilia com o git, o índice de tarefas e o código — e nunca inventa uma conexão.

Exemplo
/mss-spec:mapa
  → atualiza "onde estamos / próximo passo" e confirma as conexões do projeto
lido na partidaconexões entre projetos
C19mapa-neuralmapa mental do projeto

Desenha o projeto como um mapa mental: o projeto no centro e tudo que gira em volta, pra bater o olho e entender o todo.

Lê o próprio repositório e monta quatro ramos — arquitetura (camadas e arquivos), APIs & integrações, memórias & conhecimento (specs, decisões, aprendizados) e conexões entre projetos. Gera dois arquivos em docs/: um índice em texto (que o assistente consulta sem reabrir o projeto inteiro) e uma página interativa — clique pra expandir, arraste, passe o mouse pra ver nome/local/resumo de cada peça. Arquivo único, sem depender de internet.

Exemplo

  → docs/mapa-neural.html (mapa interativo) + docs/mapa-neural.md (índice de texto)

05 — Spec-Driven

Combinar antes de codar

O coração do kit é o Spec-Driven: antes de escrever código, escreve-se em poucas linhas o que a funcionalidade precisa fazer. Isso vira um contrato curto — e o código nasce guiado por ele, com menos retrabalho.

No nosso exemplo, quando você roda /mss-spec:nova-feature "disparo no whatsapp", o assistente não sai codando. Primeiro ele alinha com você uma spec assim:

# disparo-whatsapp — spec viva

## Objetivo
Ao cadastrar um cliente com WhatsApp válido, enviar uma mensagem
de boas-vindas automática.

## Critérios de aceite
- DADO um cliente novo com WhatsApp válido,
  QUANDO salvo o cadastro,
  ENTÃO uma mensagem é disparada e o status fica "enviado".
- DADO uma falha no envio,
  ENTÃO o erro é registrado no log (sem gravar o número) e
  o status fica "falhou".

## Fora de escopo
Disparo em massa/agendado (fica pra outra feature).

Só depois do seu OK nessa spec é que vêm o plano e o código (com teste primeiro). E o mais importante: essa spec fica viva.

Spec viva = nunca mente
A spec fica guardada em docs/specs/<assunto>.md com duas partes: Estado atual (como a coisa se comporta hoje) e Histórico (uma linha por mudança). Se amanhã alguém mexer no disparo, atualiza a spec junto — assim ela nunca vira uma mentira que induz o assistente a reintroduzir um bug. É a diferença entre "documentação que envelhece" e "contrato que acompanha o código".
Bugfix e ajuste pequeno não precisam disso
Corrigir um bug não exige a cerimônia toda: o teste que reproduz o problema já é a "especificação". O fluxo completo é para funcionalidade nova. (E o quanto de ritual você regula com o modo.)
Um assunto por janela
Cada conversa trata de um assunto só. Se, no meio de uma tarefa, surgir um tema diferente, o assistente alerta em vez de emendar — sugere fechar o que está aberto, anotar o novo tema no to-dolist e abrir uma nova conversa para ele. É um aviso, não uma trava: se você mandar seguir mesmo assim, ele segue. Assim cada branch e cada spec ficam limpas, com um foco só.
06 — O que o kit lembra

09 — Segurança

Segurança & segredos, sem susto

Como os apps ficam expostos na internet, o kit trata segurança como parte do trabalho, não como enfeite — o princípio é “seguro por padrão” (Security by Default): tudo nasce fechado e só se abre o necessário. Três pontos que qualquer pessoa entende:

  • Esconder não é proteger. Renomear ou "embaralhar" um endereço não protege nada — quem quer, acha. A defesa de verdade é o servidor conferir a permissão a cada pedido.
  • Segredo fica fora do código. Senhas moram numa variável de ambiente — no arquivo local .env (que nunca é compartilhado) em desenvolvimento, e nas configurações da nuvem (Azure App Settings) em produção. Nunca dentro do código que vai pro repositório.
  • Dado pessoal não vai pro log. No nosso exemplo, o serviço de WhatsApp registra "enviado" ou "falhou" — nunca o número, o e-mail ou o nome do cliente (regra de LGPD embutida no log).

Sistemas que conversam entre si usam uma "senha de crachá" (Bearer token), guardada só no servidor. O login de usuário final fica para uma etapa futura.

10 — Redes de segurança

O que te protege quando algo dá errado

Além de ajudar a construir, o kit tem três redes prontas para o momento em que algo sai do esperado. Não são comandos novos — já estão no lugar; esta seção só as deixa visíveis.

  • O próprio kit se testa. O mss-spec tem uma bateria de testes de smoke — rode pytest tests/ — que confere que todo caminho citado nos comandos e moldes existe de verdade, que os manifestos são válidos e coerentes. É a garantia de que uma edição num comando não deixou uma referência morta para trás. (Um framework que prega TDD para os projetos testa também a si mesmo.)
  • O git é o rollback. Não há comando de "desfazer" dedicado — e é de propósito. Como o kickoff e o upgrade só mexem em arquivos versionados, com a árvore de trabalho limpa, um git restore (ou descartar a branch) já reverte tudo. O git que você já usa é a rede de reversão; criar um comando só para isso seria peso sem ganho.
  • O CHANGELOG é a rede contra drift. Toda mudança relevante entra no CHANGELOG.md versionado, com o número da versão. É o que evita o drift entre cópias — o risco de dois colegas rodarem "o mesmo v0.8" que na verdade divergem. Na via git, o marketplace update já traz a versão nova; na via pasta local, é o CHANGELOG que diz se uma cópia está atrasada.
Desfazer um kickoff ou um upgrade
Árvore de trabalho limpa antes de rodar? Então git restore . desfaz as mudanças de arquivo, ou você simplesmente descarta a branch. Nenhum dos dois toca em código seu fora do que está versionado.

O que ainda vem

  • CI no GitHub. O repositório já está publicado no GitHub e a distribuição por URL funciona; o .gitlab-ci.yml existente é do GitLab e não roda no GitHub — um equivalente em GitHub Actions (.github/workflows/) fica como próximo passo.
  • Documentar a distribuição. Manter o histórico de mudanças (CHANGELOG) e esta página em dia a cada novidade.

GL — Glossário

As palavras técnicas, em português claro

Toda palavra técnica no texto acima é um link que traz você até aqui. Definições curtas, sem jargão:

Claude Code
O programa de inteligência artificial da Anthropic que roda no seu computador e consegue ler/escrever arquivos e rodar comandos. O mss-spec trabalha dentro dele. (No chat comum do site não há esse acesso a arquivos.)
Plugin
Um "acessório" que você instala uma vez e que adiciona recursos novos (aqui, comandos e moldes) ao Claude Code.
superpowers
Outro plugin (oficial da Anthropic) que traz a disciplina de trabalho: não sair codando sem alinhar, testar, verificar, revisar. O mss-spec depende dele (habilitado à parte).
CRUD
As quatro operações básicas de um cadastro: Create, Read, Update, Delete (criar, ler, atualizar, apagar). O "cadastro de clientes" do exemplo é um CRUD.
Spec-Driven / spec viva
"Guiado por especificação". Antes de programar, escreve-se em poucas linhas o que a funcionalidade precisa fazer; o código vem depois, guiado por isso. A spec fica "viva" — atualizada junto com o código, pra nunca mentir. Menos retrabalho.
Comando (slash command)
Um atalho que você digita começando com barra, como /mss-spec:kickoff, para disparar uma ação. Roda porque você pediu.
Skill (sugestão automática)
Uma instrução que o Claude Code ativa sozinho quando o assunto casa — ao contrário do comando, que você digita. Útil como rede de segurança, mas pode não disparar.
Marketplace
A "loja" de onde o Claude Code instala plugins. Pode ser um repositório git (adicionado por URL, que o Claude Code clona) ou uma pasta local — as duas vias do mss-spec (a lojinha se chama mss-local).
Scaffolding
"Andaime". Gerar automaticamente os arquivos iniciais de um projeto a partir de moldes, em vez de copiar tudo à mão.
TDD / teste
"Desenvolvimento guiado por testes". Escreve-se primeiro um teste que descreve o resultado esperado (e falha), depois o código que faz o teste passar. Assim, se algo quebrar depois, o teste avisa.
Verificação
Provar que funciona rodando de verdade e mostrando o resultado — em vez de só "confiar que deu certo".
Code review
Revisão do código por outro olhar (aqui, um agente revisor) antes de considerar pronto.
Log
O "diário de bordo" do app: mensagens que registram o que aconteceu (e o que deu errado), pra diagnosticar problemas. Nunca deve conter senha nem dado pessoal.
Git
O sistema que guarda o histórico do código: quem mudou o quê e quando, com a possibilidade de voltar atrás. O "repositório" é o projeto guardado no git.
Branch
Uma "linha paralela" de trabalho no git. Você mexe nela sem bagunçar a versão principal e depois junta. O kit abre uma branch por tarefa, sempre a partir da linha principal (nunca de outra branch), para o ponto de partida ser sempre previsível.
Docker / container
Uma forma de empacotar o app com tudo que ele precisa numa "caixa" isolada (o container) que roda igual em qualquer máquina.
Proxy
O "porteiro" de rede da empresa: todo acesso à internet passa por ele. Precisa ser configurado para o app conseguir baixar coisas.
CA / certificado
O "carimbo" que prova que uma conexão segura (HTTPS) é confiável. A empresa tem o seu, que precisa entrar na caixa (container) para as conexões funcionarem.
Postgres / SQL Server
Dois tipos de banco de dados — onde o app guarda a informação. A MSIG usa os dois, conforme o projeto.
Azure (App Settings)
A nuvem da Microsoft onde os apps da MSIG são publicados. "App Settings" é onde se guardam configurações e segredos com segurança, lá na nuvem. É de lá que os logs de produção são capturados.
.env / variável de ambiente
Um lugar fora do código para guardar configurações e senhas. O arquivo .env é local e nunca é compartilhado; na nuvem, o mesmo papel é feito pelas configurações do Azure.
Fernet
Um jeito de embaralhar (criptografar) um texto. Só é segurança de verdade se a "chave" ficar guardada em outro lugar; se ela viaja junto do código, é apenas um disfarce.
OWASP
Uma lista famosa dos riscos de segurança mais comuns em aplicações web — usada como checklist para não esquecer o básico.
Bearer token
Uma "senha de crachá" que um sistema apresenta a outro para provar que tem permissão de chamar. Fica guardada só no servidor.
Front-end / React / Mantine
A parte visual do app (as telas). React e Mantine são ferramentas para construir telas mais ricas e interativas.
